# 统一流式播放优化

## 修改日期
2026-04-19

## 问题描述

### 问题1：小文件先下载再播放机制导致用户体验差
原有的实现中，小于50MB的文件会先完整下载到本地临时文件后再开始播放。这导致：
- 用户需要等待完整的下载时间才能听到声音
- 在网络慢的情况下，等待时间可能很长
- 浪费存储空间（每个文件都需要完整保存）
- 与大文件的流式播放体验不一致

### 问题2：载入播放列表时界面失去响应
当从SSH服务器加载包含大量文件的目录或播放列表时：
- SSH的`listDirectory`操作是同步阻塞的
- 如果网络慢或文件数量多，UI线程会被长时间阻塞
- Android系统可能判定为ANR（Application Not Responding）并强制关闭应用
- 用户看到界面卡死，无法进行任何操作

## 解决方案

### 方案1：统一使用流式播放

**核心改动：**
1. 移除`_playMediaAfterDownload()`方法（小文件下载后播放）
2. 所有文件（无论大小）都通过`_playMediaStreaming()`方法进行流式播放
3. 取消预下载功能（因为流式播放不需要预下载）

**技术实现：**
```dart
// 之前：根据文件大小选择不同播放方式
if (sizeInMB > 50) {
  await _playMediaStreaming(file);  // 大文件：流式
} else {
  await _playMediaAfterDownload(file);  // 小文件：下载后播放
}

// 现在：所有文件统一使用流式播放
await _playMediaStreaming(file);  // 所有文件：流式
```

**优势：**
- ✅ 用户可以立即开始播放，无需等待下载完成
- ✅ 边下边播，节省存储空间
- ✅ 统一的播放体验，无论文件大小
- ✅ 减少代码复杂度，移除不必要的分支逻辑
- ✅ 降低内存占用（不需要缓存完整文件）

### 方案2：异步加载播放列表和目录

**核心改动：**
1. 在`addDirectoryToPlaylist()`中使用分批处理和异步让出
2. 在`loadPlaylist()`中使用分批加载
3. 在`_loadCurrentDirectory()`中添加异步让出点

**优势：**
- ✅ UI线程不会被长时间阻塞，保持响应
- ✅ 用户可以实时看到加载进度
- ✅ 避免Android系统的ANR检测
- ✅ 提升用户体验，界面流畅不卡顿
- ✅ 支持超大播放列表（数千首歌曲）

## 技术要点

### 1. 流式播放原理
流式播放通过HTTP服务器将SSH远程文件转换为本地HTTP流：
```
SSH Server → Streaming Service (HTTP Proxy) → Audio Player
     ↓                    ↓                        ↓
  远程文件          边读边转发                边下边播
```

### 2. 异步让出机制
Dart的`Future.delayed(Duration.zero)`是一个常用的异步让出技巧：
```dart
await Future.delayed(Duration.zero);
```

**作用：**
- 将当前任务放入事件队列末尾
- 让出控制权给其他微任务（包括UI渲染）
- 不会引入实际延迟（Duration.zero）
- 确保UI线程有机会处理用户输入和绘制

### 3. 分批处理策略
对于大量数据的处理，采用分批策略：
- **目录加载**: 不分批（通常文件数不多）
- **播放列表添加**: 每批50个文件
- **播放列表加载**: 每批100个文件

### 修改的文件

## 注意事项

### 1. 网络依赖性
流式播放强依赖网络连接质量：
- 网络中断会导致播放中断
- 建议在WiFi环境下使用

### 2. 后台播放
流式播放需要在后台保持：
- HTTP服务器必须持续运行
- SSH连接必须保持活跃
- 前台服务通知不能被关闭
