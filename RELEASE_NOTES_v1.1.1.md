# v1.1.1 版本更新说明

## 修复内容

### 核心问题
在 v1.1.0 版本中，断网续播功能与用户主动暂停逻辑存在冲突，导致：
- 后台自动恢复播放基本失效
- 前台恢复播放工作不正常
- 车机 Play/Pause 键控制与断网续播相互干扰

### 根本原因
`_userManuallyPaused` 标志在多个地方被错误地清除，导致无法准确判断用户的真实意图。

## 解决方案

### 核心原则
**保持 `_userManuallyPaused` 标志的稳定性**，只在用户明确操作时修改，不在自动恢复过程中清除。

## 逻辑说明

### 用户主动暂停的场景
以下操作会设置 `_userManuallyPaused = true`：
- UI 点击暂停按钮
- 通知栏点击暂停按钮
- 车机 Play/Pause 键（当前为播放状态）

**行为**：无论网络是否断开或恢复，都不会自动恢复播放，等待用户手动操作。

### 系统强制暂停的场景
以下操作**不改变** `_userManuallyPaused`：
- 音频焦点丢失（其他应用占用音频）
- 电话呼入
- SSH 断开时的状态保存

**行为**：如果此时断网，网络恢复后会自动断点续播。

### 正在播放时断网的场景
- `_userManuallyPaused = false`（因为正在播放）
- SSH 断开 → 保存播放进度
- 网络恢复 → 自动恢复到断网前的位置

**行为**：前台和后台都能正常断点续播。

## 测试场景

### ✅ 场景 1：用户主动暂停后断网
1. 用户点击暂停 → `_userManuallyPaused = true`
2. 网络断开 → 不触发自动恢复
3. 网络恢复 → 不自动恢复播放
4. **结果**：保持暂停状态 ✅

### ✅ 场景 2：正在播放时断网
1. 正在播放 → `_userManuallyPaused = false`
2. 网络断开 → 保存进度，设置恢复标志
3. 网络恢复 → 自动断点续播
4. **结果**：恢复到断网前的位置 ✅

### ✅ 场景 3：音频焦点丢失后断网
1. 音频焦点丢失 → `pauseBySystem()` 不改变标志
2. 网络断开 → 正常触发自动恢复
3. 网络恢复 → 自动恢复播放
4. **结果**：自动恢复播放 ✅

### ✅ 场景 4：车机 Play/Pause 键切换
1. 播放中按 Pause → `_userManuallyPaused = true`
2. 再按 Play → `_userManuallyPaused = false`，开始播放
3. **结果**：正常切换，互不干扰 ✅
