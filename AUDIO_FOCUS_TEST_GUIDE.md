# 音频焦点丢失Bug修复 - 测试指南

## 修复内容

本次修复解决了以下问题:
- **Bug**: 用户暂停播放后,其他应用获取音频焦点并播放,退出后本app自动恢复播放
- **原因**: just_audio的audio_session在重新获得音频焦点时可能自动恢复播放,且网络状态变化也可能触发恢复逻辑
- **方案**: 添加`_audioFocusLost`标志,在所有自动恢复播放的地方进行检查

## 修改文件

1. `lib/providers/app_provider.dart` - 添加_audioFocusLost标志及相关检查
2. `lib/main.dart` - 在媒体控制监听器中添加检查
3. `AUDIO_FOCUS_AUTO_PLAY_FIX.md` - 详细修复文档

## 调试日志关键词

在测试过程中,可以查看logcat日志来验证修复是否生效:

```bash
adb logcat | grep -E "音频焦点|audioFocusLost|系统强制暂停|用户主动暂停"
```

**关键日志:**
- `⏸️ 系统强制暂停，_isPlaying = false, _audioFocusLost = true` - 音频焦点丢失时设置标志
- `⚠️ 音频焦点已丢失，忽略自动播放命令（需用户手动恢复）` - 阻止自动播放
- `▶️ 用户主动播放，_isPlaying = true, _userManuallyPaused = false, _audioFocusLost = false` - 用户手动恢复时清除标志

## 常见问题

### Q1: 为什么有时候还是会自动播放?
A: 如果SSH断开时用户**没有**主动暂停,SSH重连成功后会自动恢复播放。这是预期行为,因为用户可能只是网络临时中断,希望继续收听。

### Q2: 如何区分"用户主动暂停"和"系统强制暂停"?
A: 
- 用户主动暂停: `_userManuallyPaused = true`, 用户点击了暂停按钮
- 系统强制暂停: `_audioFocusLost = true`, 因音频焦点丢失、来电等系统事件导致

两种情况都会阻止自动恢复播放,但标记不同以便调试和日志追踪。

### Q3: 如果我想在音频焦点恢复后自动继续播放怎么办?
A: 当前的设计是**不自动恢复**,需要用户手动操作。这是为了避免打扰用户(例如用户正在使用其他应用听声音)。如果需要自动恢复,可以移除`_audioFocusLost`相关的检查,但这可能会导致用户体验问题。

- 音频焦点丢失后不应自动恢复
- SSH/网络断开后用户暂停,恢复后不应自动播放
- 用户通过通知栏/蓝牙设备点击播放应正常播放
